# Synthesis

## Design-space statement

Question:

1. What missing branches should the immigration project explore beyond the current fiscal core, especially around crime, laws, politics, and downstream effects?

Current approach being escaped:

1. repeated return to fiscal scalars, housing debate, and broad “local burden” rhetoric

Hard constraints:

1. no fake precision
2. no category slip
3. no vague cohesion talk without mechanism
4. prefer modules that can connect to current datasets or plausible near-term acquisitions

## Explore

1. `I01` `Court and interpreter friction module`
   Why: this is a real downstream burden already documented in the repo, but still trapped inside narrative memo prose.
2. `I02` `School service complexity index`
   Why: it is the cleanest upgrade over the repo’s current blunt school-spending proxies.
3. `I03` `Labor-standards erosion module`
   Why: this opens a non-fiscal but still measurable branch that changes how “immigrant cost” should be attributed between employers, workers, and the state.
4. `I06` `Institutional overload scorecard`
   Why: this converts “institutional resilience” from a framing claim into an operating model with thresholds and triggers.
5. `I07` `Stress-test scenario engine for local destinations`
   Why: the repo already knows averages are weak. Stress testing makes that explicit.
6. `I09` `Political feedback module`
   Why: politics is already in the repo, but still mostly as interpretation. This would turn it into a downstream state-capacity variable.
7. `I11` `Admission-channel composition layer`
   Why: it is a direct threat to overreading origin-level findings as cultural or fixed-group facts.

## Parked

1. `I04` `Legal-regime delta ledger`
   Reason: strong idea, but better after the operational modules exist.
2. `I05` `Crime channel split beyond offending rates`
   Reason: worth revisiting once the repo decides whether it wants a full justice-system branch.
3. `I08` `Native adaptation and exit module`
   Reason: important but data-intensive; likely later than courts, schools, and stress testing.
4. `I10` `Assimilation velocity module`
   Reason: long-run valuable, but the current public stack is still too weak for a high-confidence lifecycle branch.

## Rejected

1. `I12` `Housing code-enforcement branch as a standalone project`
   Reason: housing is already one of the most explored branches. It is not the highest-value next marginal move.

## Paradigm gaps surfaced

1. The repo still lacks a true `operations` layer for institutions.
2. The repo still lacks a clean way to model `nonlinear overload` rather than average burden.
3. The repo still risks overreading `origin effects` that may actually be `admission-channel effects`.
4. The repo discusses politics and backlash, but not yet in a way that is structurally comparable to school or housing modules.

## Best next step

If only one branch is built next, it should be:

1. `I02` plus `I06`

That combination would do the most to move the project from “school pressure exists” to “here is how local educational systems move from absorbable load to overload.”

If two more are added after that:

1. `I01`
2. `I11`

That would create a tighter downstream institutional map:

1. schools
2. courts / interpreter systems
3. admission-channel composition

## Bridge to action

Brainstorm identified `12` ideas, with `7` worth exploring and `7` surviving the pain-point gate.

Best follow-up options:

1. plan the top `1-2` modules
2. run `/model-review` on `I02 + I06` as the most plausible next tranche
