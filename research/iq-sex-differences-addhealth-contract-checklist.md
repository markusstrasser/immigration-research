1. start the application in the `CPC Data Portal`
2. add the required datasets to cart
3. submit the basic portal application
4. complete the contract attachments and security documents

[INFERENCE] The important practical point is that the scientific request and the contract paperwork are coupled. The request should therefore stay narrow enough that `Attachment B`, the IRB language, and the security plan all remain coherent.

## IRB Checklist

The contract language says the project needs an IRB-signed approval document covering the research project and the secure use / handling of the requested data. [SOURCE: https://data.cpc.unc.edu/docs/Restricted-Use_Contract_-_New.pdf]

That means the IRB submission should explicitly mention:

1. use of `Add Health` restricted-use data
2. transcript-linked educational records from `AHAA`
3. remote secure-workspace access if that is the access mode used
4. the specific research team members who will access the data

[INFERENCE]

Do **not** submit a generic IRB covering “secondary analysis of educational data” if it does not mention restricted transcript-linked Add Health data and secure handling. [INFERENCE]

## Security Checklist

The current UNC security materials say new contracts use the `Secure Research Workspace` model and require a security plan and signed security pledges. [SOURCE: https://data.cpc.unc.edu/docs/Security_Information_for_Remote_Add_Health_Access.pdf; https://data.cpc.unc.edu/docs/Security_Plans_for_Restricted-Use_Data_Information.pdf]

### Before submission

1. decide who will actually access the data
2. decide from which physical locations they will access it
3. decide whether any IT staff need to be named
4. make sure the named workflow matches the actual planned workflow

[INFERENCE]

The instructions packet says `Location` is required on the security pledge forms. [SOURCE: https://data.cpc.unc.edu/docs/Information_Packet_-_Restricted-Use__-_NEW_2025-01-15.pdf]

Do **not** name extra researchers “just in case.” Every direct-access person increases paperwork and security obligations. [INFERENCE]

## Recommended Filing Order

1. confirm the portal still exposes the exact `AHAA` files needed
2. submit the narrow data cart
3. download the actual forms generated by the portal
4. fill `Attachment B` with the scope and crosswalk docs in this repo
5. draft the IRB language against the exact requested files
6. complete the security plan around the real remote-workflow setup
7. only then expand the request if the portal makes a bundled transcript package obviously cleaner

[INFERENCE]

## Failure Modes To Avoid

1. requesting too many `AHAA` files at once and making the justification bloated
2. writing the project as a broad ideological “sex differences” study instead of a transcript/test school-mechanism study
3. naming more direct-access researchers than necessary
4. drafting the IRB too generically
5. assuming old contract PDFs override the current portal workflow
